我的 Mihomo DNS 写法：国内快，国外稳，尽量避开污染

本文基于我自己的 echo.yaml 配置整理。我的使用场景很明确：人在大陆，国内网站要快，国外网站、AI 服务、GitHub、Google、YouTube 这类站点也要稳定，并且尽量减少 DNS 污染对访问结果的影响。

这套配置不是为了追求某个 DNS 跑分第一，而是让 DNS、代理规则、fake-ip 模式配合起来，做到日常使用稳定、逻辑清晰、后续维护简单。

我的目标

  1. 国内域名优先使用国内加密 DNS，减少解析绕路。
  2. 国外和容易被污染的域名交给海外加密 DNS 兜底。
  3. 使用 fake-ip 模式，让 Mihomo 根据规则接管后续连接。
  4. 不在 nameserver-policy 里堆大量重复域名，保持配置可维护。
  5. 保留原来的 fake-ip-filter，避免影响局域网、NTP、系统连通性检测等场景。

核心配置

下面是我 DNS 段的核心写法。实际配置里 fake-ip-filter 还有一长串域名，我保留原列表不动，这里只展示 DNS 决策相关的主体部分。

为什么这样写

1. DNS 只监听本机

我的配置里 allow-lan: false，也就是不打算给局域网其他设备使用。因此 DNS 监听在 127.0.0.1 更合适，只服务本机。

如果后面要把这台机器当作局域网网关，给手机、平板或其他电脑使用，再改成：

2. 继续使用 fake-ip

fake-ip 模式下，命令行里看到域名解析成 198.18.x.x 是正常现象，不代表 DNS 坏了。这个地址段是 Mihomo 用来做连接映射的，真正访问时会结合代理规则、嗅探和连接目标继续处理。

所以在这种模式下，不要只用 nslookup 或 Resolve-DnsName 的结果判断 DNS 好坏。更可靠的判断方式是看 Mihomo DNS 日志和实际访问结果。

3. default-nameserver 用国内公共 DNS 做启动解析

default-nameserver 主要用于解析 DoH/DoT 服务器本身的域名，也就是启动阶段的基础解析。这里我选择阿里 DNS 和 DNSPod 的国内公共 DNS，因为它们在大陆网络里通常更近、更容易连通。

我没有把 114.114.114.114 放在主解析里。它在很多网络里确实很快，但它是传统明文 DNS，不适合作为这套“尽量减少污染和劫持”的配置核心。

4. 国内解析走阿里 DNS 和 DNSPod 的加密 DNS

nameserver 是日常主解析。我这里没有使用单一服务，而是同时配置阿里 DNS 和 DNSPod，并且同时使用 DoH 和 DoT。

这样做的原因很简单：

  1. 国内域名用国内 DNS，解析结果通常更适合大陆网络。
  2. DoH/DoT 是加密 DNS，比普通 UDP DNS 更不容易被中间链路篡改。
  3. 不单押一家服务，某一家偶发慢或者不可用时还有其他选择。

需要注意的是，公共免费 DNS 服务不等于企业级 SLA。比如阿里云公共 DNS 免费版官方就说明过免费版可能存在限速和服务保障限制，所以个人使用可以这样搭配，但高并发业务不要直接照搬。

5. 国外和敏感域名交给 fallback

fallback 里我放的是 Cloudflare 和 Google 的加密 DNS。它们更适合处理海外域名，尤其是 Google、YouTube、GitHub、OpenAI、Claude 这类服务。

这里同样没有只写一个服务，而是 Cloudflare 和 Google 都放进去。原因还是稳定性：公共 DNS 服务本身也会受本地网络、运营商、时间段影响，多放几个可信 resolver 比单押一个更稳。

6. fallback-filter 决定什么时候用海外结果

fallback-filter 是这套配置里很关键的一层。它不是简单地让所有域名都走海外 DNS，而是只在需要的时候使用 fallback。

我的思路是：

  1. 国内站点尽量保留国内 DNS 结果。
  2. 命中 geosite:gfw 的域名使用 fallback。
  3. Google、YouTube、OpenAI、Claude、GitHub 等明确容易被污染或需要代理访问的域名，直接列入 fallback 域名列表。
  4. 240.0.0.0/4 这类异常地址段作为过滤条件，避免明显不正常的解析结果。

这样国内网站不会因为海外 DNS 解析而绕路，国外服务也不会因为国内 DNS 污染拿到错误结果。

7. nameserver-policy 只保留大方向

很多配置会在 nameserver-policy 里堆大量域名，比如每个 AI 网站、每个 GitHub 相关域名都单独写一遍。这样短期看起来很精细，但后期维护会越来越麻烦。

我这里把策略收敛成一条：

国内和私有域名明确走国内加密 DNS。国外和敏感域名则通过 fallback-filter 去判断。这样配置更短，也更容易看懂。

8. fake-ip-filter 不要乱删

fake-ip-filter 主要用于告诉 Mihomo：哪些域名不要返回 fake-ip，而是保留真实 IP。

我的配置里保留了局域网、NTP、时间同步、系统检测、部分游戏和服务发现相关域名。它们如果被 fake-ip 接管，可能会出现时间同步失败、局域网设备发现异常、系统联网检测不准确等问题。

所以优化 DNS 时，我只替换 DNS 决策层，不动原来的 fake-ip-filter 列表。

我不推荐的写法

1. 所有域名都走海外 DNS

这样看起来“干净”，但国内网站可能拿到不适合大陆网络的解析结果，访问反而变慢。

2. 所有域名都走国内普通 DNS

国内网站通常没问题，但 Google、GitHub、YouTube、OpenAI 这类域名更容易受到污染或错误解析影响。

3. 用命令行解析结果直接判断 fake-ip 配置坏了

在 fake-ip 模式下，看到 198.18.x.x 是预期行为。最终要看浏览器访问、代理规则命中情况和 Mihomo 日志。

4. nameserver-policy 写得过细

过细的域名策略短期能用，长期容易变成维护负担。我的做法是：国内和私有域名用 nameserver-policy 定方向，国外敏感域名用 fallback-filter 兜底。

验证方式

重载配置后，我一般看这几类结果：

  1. Mihomo 或 Clash 客户端没有 DNS 字段报错。
  2. DNS 日志里，国内域名优先使用 dns.alidns.com 或 doh.pub。
  3. Google、OpenAI、GitHub、YouTube 等域名命中 fallback。
  4. baidu.com、bilibili.com、taobao.com 访问保持快速。
  5. openai.com、chatgpt.com、github.com、google.com、youtube.com 能稳定走代理规则。

如果命令行里看到 198.18.x.x，先不要急着改配置。fake-ip 模式下这很正常，重点看实际访问和客户端日志。

总结

这套 DNS 配置的核心不是“哪个 DNS 最快”，而是分工：

  1. 国内域名：国内加密 DNS，优先速度和本地化解析。
  2. 国外敏感域名：海外加密 DNS fallback，优先稳定和避免污染。
  3. 代理判断：交给 Mihomo 的规则和 fake-ip 机制处理。
  4. 配置维护：少写重复策略，保留必要的 fake-ip-filter。

对我这种大陆日常使用环境来说，这种写法比单纯堆 DNS 地址更稳定，也更容易长期维护。
